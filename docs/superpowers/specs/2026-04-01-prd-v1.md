# DBS Store — PRD v1

## Vision & Scope

**Produit :** DBS Store — boutique en ligne d'électronique pour la Côte d'Ivoire / zone UEMOA.

**Cible :** Consommateurs ivoiriens francophones cherchant des produits tech (smartphones, tablettes, ordinateurs, audio, accessoires).

**Objectif v1 :** Un e-commerce fonctionnel de bout en bout — du catalogue à la commande livrée — avec un back-office admin complet. Paiement à la livraison (COD) uniquement.

**Hors scope v1 :**
- Paiement en ligne (Mobile Money, carte bancaire)
- Multi-langue
- App mobile
- Programme de fidélité
- Avis/notes produits

## Architecture technique

### Stack

| Couche | Technologie |
|--------|-------------|
| Frontend | Next.js 16, React 19, TypeScript |
| UI | Tailwind CSS v4, Shadcn UI (Radix), Lucide icons |
| State | Zustand (panier client) |
| Auth | Better Auth (email/password, social, OTP, RBAC orga) |
| Base de données | Cloudflare D1 (SQLite) via Drizzle ORM |
| Stockage fichiers | Cloudflare R2 (presigned URLs via AWS SDK) |
| Cache | Cloudflare KV (sessions, navigation) |
| Jobs async | Cloudflare Queue (emails, image resize) |
| Email | Resend |
| Deploy | Cloudflare Workers via @opennextjs/cloudflare |
| Tests | Vitest + React Testing Library + jsdom |

### Schéma de données

```
products        — catalogue (prix en FCFA entier, images/specs en JSON)
orders          — commandes (statut, livraison, paiement COD)
order_items     — lignes de commande (snapshot prix au moment de l'achat)
hero_slides     — bannières carousel (ordonnées, max 5 actives)
categories      — table D1 avec hiérarchie parent/enfant
user/session/*  — tables Better Auth (auth, orga, RBAC)
newsletter      — inscriptions newsletter (email, date)
```

### Flux principal

```
Visiteur → catalogue → fiche produit → ajouter au panier (Zustand/localStorage)
→ connexion/inscription → vérification email → checkout (COD)
→ commande créée en DB → confirmation → suivi dans /compte/commandes
→ admin gère le statut (pending → confirmed → shipped → delivered)
```

## Exigences non-fonctionnelles

### Performance
- First Contentful Paint < 2s (SSR/RSC pour le storefront)
- Images optimisées (WebP, lazy loading)
- Cache KV pour la navigation et les sessions

### Sécurité
- Routes admin protégées par middleware (auth + email vérifié + membre orga)
- Routes compte protégées (auth + email vérifié)
- Parameterized queries (pas de concaténation SQL)
- Presigned URLs scopées par bucket/clé avec expiration
- Validation des inputs côté serveur (server actions)
- RBAC : owner > admin > member

### Fiabilité
- CI obligatoire avant merge (lint + tests + build)
- Branch protection sur main
- Transactions D1 batch pour les écritures atomiques

### Localisation
- Français uniquement (v1)
- Prix en FCFA (entier, pas de décimales)
- Zone UEMOA

### SEO
- SSR pour toutes les pages publiques
- Metadata dynamiques (titre, description par produit/catégorie)
- Sitemap et robots.txt

## Workflow de développement

### Pipeline CI/CD

**Sur PR vers main :**
```
bun run lint → bun run test → bun run build
```

**Sur merge dans main :**
```
bun run lint → bun run test → bun run build → wrangler d1 migrations apply → wrangler deploy
```

**Protection de branche main :**
- PR obligatoire (pas de push direct)
- CI doit passer avant merge
- Squash merge uniquement

### Cycle de vie d'une feature

```
1. Issue GitHub         — créer depuis le PRD, label par feature
2. Brainstorm           — si la feature est non triviale
3. Plan                 — pour les features > 1 fichier
4. Branche              — feat/<nom> depuis main
5. Développement        — code + tests (TDD pour la logique métier)
6. Validation locale    — bun run test + bun run lint + bun run build
7. PR                   — description claire, lien vers l'issue
8. Review auto          — CI + review Claude (pr-review-toolkit)
9. Review CodeRabbit    — review automatique via CodeRabbit sur chaque PR
10. Security review      — sur les PRs touchant auth/admin/paiement/S3
11. Merge squash        — ferme l'issue (Fixes #XX)
12. Deploy auto         — CI déploie sur Cloudflare
```

### Qualité

- **Review stricte** : tests + lint + review Claude + CodeRabbit + security review sur les parties sensibles
- **Security review déclenchée** quand la PR modifie : `lib/auth/`, `lib/actions/`, `middleware.ts`, `app/api/`, `app/(admin)/`

### Gestion produit

- **Source de vérité** : ce PRD pour la vision v1, issues GitHub pour le suivi
- **Labels** : `type:feature`, `type:bugfix`, `type:refactor`, `type:infra`, `priority:high/medium/low`, `security`
- **Bugs bloquants** : traités avant les features en cours
- **Bugs mineurs** : backlog, traités entre deux features
